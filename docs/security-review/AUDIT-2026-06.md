# Recipe App — Post-Remediation Security & Functional Audit (2026-06)

**Audit dates:** 2026-06-28 (Phases 1–2) → 2026-07-15 (Phases 3–6)
**Scope:** Independent re-review of the backend (Go/Gin/GORM/Postgres) and frontend
(React/TypeScript), following the `remediation-with-vpn` plan (PRs #31–#38). Read-only —
no production code was modified anywhere in this audit; every phase file under
`docs/security-review/audit-2026-06/` states this explicitly.
**Threat model:** The app launches behind a small, trusted-few Tailscale VPN. A **compromised
or malicious tailnet member is in scope** even in the with-VPN launch. The VPN will be
**removed later** for public-internet exposure — findings are classified against both states.

**Source material:** every file in `docs/security-review/audit-2026-06/` — `00-notes.md`,
`01-baseline.md`, `01-deps.md`, `01-routes.md`, `02-ai.md`, `02-atomicity.md`, `02-authz.md`,

## 1. Executive summary

Phase 2 verification of the prior remediation work (PRs #31–#38) came back **PASS on 10 of 11
checks** — SSRF hardening, JWT fail-closed boot, upload/path-traversal lockdown, transactional
atomicity, cross-user authorization, AI prompt-injection hardening, and logout session-clearing
all hold up under live re-testing. The one FAIL (`02-secrets.md`) is serious: a real,
**unrotated** Anthropic API key is still recoverable from local git history.

Phases 3–4's fresh, independent re-review of current code found **no new IDOR, no SQL
injection, no XSS, and no bypass** of the hardened SSRF/upload/prompt-injection surfaces — the
remediation work is holding. What it did find is a set of **perimeter-dependent controls**
(rate-limiting, email verification, revocation, security headers) that are today invisible
behind the VPN and become real gaps the moment it's removed, plus **two live functional
breakages** unrelated to the VPN: registration is broken end-to-end, and every configured AI
model ID has passed its retirement date.

**Verdict: NO-GO behind the VPN until four blockers are cleared** (Section 2). None of the four
are VPN-perimeter-dependent — they are broken today, member or no member, VPN or no VPN.
Everything else in this report is real but is either already-passing remediation work or a
`BEFORE-VPN-REMOVAL` item appropriate for the existing remediation-plan backlog.

## 2. GO/NO-GO verdict

### NO-GO — 4 blockers, all fixable in isolation, none touching the passing remediation work

| # | Blocker | Evidence | Type |
|---|---|---|---|
| 1 | **Postgres port `5432` published on all host interfaces with the static password `your_password`, TLS disabled** | `docker-compose.yml:35-39`, `28`; `03-vpn-deps.md` Finding 1 | Security — member-exploitable **today** |
| 2 | **A real, unrotated Anthropic API key is recoverable from local git history** (`git stash show -p stash@{0}` → `doc/recipe-api/ai/ai create config.yml:api_key`) | `02-secrets.md` Check 2 (FAIL) | Security — credential must be rotated regardless of blast radius |
| 3 | **Self-registration is broken end-to-end** — the live `LandingPage.tsx` `RegisterView` sends `{name,email,password}`; the backend requires `first_name`/`last_name` and 400s every attempt | `04-auth.md` Finding 1, live-confirmed; cross-referenced `06-smoke.md` | Functional — blocks onboarding *if self-registration is the intended onboarding path* (see caveat) |
| 4 | **All four configured Claude model IDs are past their retirement date, including the default fallback** (`claude-3-haiku-20240307` = `ModelDefault`) | `06-model-ids.md`, via the `claude-api` skill's authoritative retirement tables | Functional — every AI-dependent feature (URL import, PDF import, instruction parsing, item categorization) 404s on any deployment with a real provider key |

**Caveat on Blocker 3:** if the real deployment plan seeds accounts by some other means (admin
creates users directly, a fixed roster of the "small trusted few") rather than through the public
registration form, this drops from a launch blocker to a severe-but-deferrable bug. State which
onboarding path is intended before treating this as go/no-go.

**Caveat on Blockers 1 and (partially) 4/DB-adjacent findings:** the security posture asserted
throughout this audit assumes `docker-compose.yml` reflects the actual production deployment.
**The repository contains no production manifest, reverse-proxy config, or TLS terminator.** If
the real deploy fronts the stack differently (e.g. a managed Postgres with no public port, an
HTTPS-terminating proxy in front of the API), Blocker 1 and the `BEFORE-VPN-REMOVAL` plaintext-HTTP
finding (Section 4, Finding 4 in `03-vpn-deps.md`) may already be mitigated at the edge —
this was **verified directly** for this audit (`docker-compose.yml` confirmed to publish `5432`
and set `POSTGRES_PASSWORD=your_password` unconditionally, with no override file present in the
repo), but the *real* deployment topology is outside this audit's read-only scope and should be
confirmed before treating the caveat as resolved.

None of the four blockers requires touching the SSRF/upload/JWT/authz/atomicity work verified
PASS in Section 3 — they are additive fixes, not rework.

### Reconciled finding — the frontend Docker image ships the Vite **dev server**, not a production build

`services/frontend/Dockerfile:11` (`CMD ["npm", "run", "dev"]`) and `docker-compose.yml:13`
(`command: npm run dev`) run Vite's dev server in what would be the production container — there
is no separate production Dockerfile/compose override in the repo (confirmed: `find` for any
`*compose*`/build-stage variant found none). This was flagged as an open build-mode question in
`03-vpn-deps.md`'s Inconclusive section and handed to Phase 4's dependency review, but
**`04-deps.md` did not pick it up** — it evaluated the `esbuild ≤0.24.2` moderate advisory
("dev-server can be probed by any website") and dismissed it as **"dev-server-only, not prod"**
without checking which mode actually ships. Reconciling the two: if the shipped container really
runs `npm run dev`, that advisory is **live, not moot** — a member's browser visiting a malicious
external site could send cross-origin requests to the dev server (reachable at the tailnet IP)
and read the response (the classic dev-server-DNS-rebinding class of bug), on top of exposing
verbose Vite error overlays and an unauthenticated HMR websocket to the whole tailnet.

- **Severity:** MEDIUM
- **Classification:** `BEFORE-VPN-REMOVAL` — requires a member's own browser to be lured to a
  malicious external site while connected to the tailnet (a two-actor scenario, not the
  compromised-member-alone case that defines the other blockers), so it does not join Section 2's
  blocker list, but it is real and easy to fix.
- **Recommended control:** Build the frontend for production (`vite build` → serve the static
  `dist/` via a minimal static server or nginx) instead of running the dev server in the shipped
  image; add a proper multi-stage `Dockerfile` for the frontend. This also resolves the moderate
  `esbuild`/`vite` advisory chain outright rather than needing the semver-major `vite@8` bump
  `04-deps.md` recommended for a different reason.

**Judgment call — JWT non-revocation:** `05-post-vpn-triage.md` itself flagged this as arguably
closer to a blocker, since the compromised-tailnet-member threat model is in-scope *today*. This
report classifies it as `BEFORE-VPN-REMOVAL`, not a blocker: the compromised member must already
have exfiltrated a live token (via a separate compromise, e.g. device theft or an XSS this audit
found no evidence of) before non-revocation matters — it's incident-response defense-in-depth for
a scenario that requires a prior compromise, not an exploit path the member can walk unaided
today. Recommend prioritizing it early in the `BEFORE-VPN-REMOVAL` backlog given the flagged
concern, but it does not block VPN launch.

**Reclassified in Phase 5:** the "dead auth pages" code-quality item is upgraded to reflect that
it masks the live registration bug (Section 2 Blocker #3) — see `05-quality-triage.md`.

  dropped by `04-deps.md`, picked back up here. Recommend a lightweight process fix for future
  audits: Phase-N findings that explicitly hand off an open question to Phase N+1 should be
  grepped for at the start of that phase, not just referenced informally.
- **No automated CI check for model-ID staleness or dependency vulnerabilities.** `06-model-ids.md`
  recommends a periodic re-verification job; `05-quality-triage.md` confirms no `.github/` CI
  wiring for `npm audit`/`govulncheck` exists today. This is the root cause of Blocker #4 — a
  model ID silently sailed past retirement with nothing to catch it — and will recur without a
  scheduled check.
- **Deployment topology is unverified.** As stated in Section 2, this entire audit's read of
  `docker-compose.yml` as "the real deployment" is an assumption, not a confirmed fact — there is
  no production manifest in the repository. Confirm before/at go-live.
- **Process finding, not a security issue:** `phase-4-recipe-post-remediation-audit`,
  `phase-5-...`, and `phase-6-...` branches (PRs #43, #44, #45) were cut from
  `phase-2-recipe-post-remediation-audit` instead of stacking on
  `phase-3-recipe-post-remediation-audit` as the plan's stacked-PR convention intended — so PRs
  #43–#45 do not contain the Phase 3 findings (`03-*.md`). This Phase 7 branch merges Phase 3's
  branch directly to synthesize the complete file set, but the **GitHub PR base chain itself
  (#43→#44→#45) still needs a follow-up fix** (re-target #43's base to `phase-3-...`, or an
  equivalent rebase) independent of this report — flagging so it isn't silently lost.
